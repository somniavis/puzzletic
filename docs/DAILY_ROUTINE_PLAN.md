# 일일 루틴 기획 초안

> 마지막 업데이트: 2026-03-20

## 개요

Petroom에 `일일 루틴` 기능을 추가한다.

- 위치: Petroom FAB 상점 버튼 위
- 진입: 버튼 클릭 시 팝업 오픈
- 구성: 하루 1세트의 목표 3개 제공
- 보상: 모든 목표 달성 후 1일 1회 수령

이 기능의 목적은 서버 사용을 최소화하면서도, 유저에게 매일 짧고 분명한 행동 목표를 제공하는 것이다.

---

## 핵심 원칙

- 루틴 진행도는 로컬에서만 관리한다.
- 서버는 진행도를 저장하지 않는다.
- 모든 목표 달성 시에만 서버리스 claim 요청을 보낸다.
- 서버는 오늘 이미 보상을 받았는지만 확인한다.
- 서버 저장 데이터는 최소화한다.

---

## 루틴 규칙

### 1. 기본 구조

- 루틴은 하루 1세트만 제공한다.
- 하루 루틴은 총 3개 목표로 구성한다.
- 목표는 `A 풀에서 1개`, `B 풀에서 2개`를 선택한다.

### 2. 목표 풀

#### A. 학습 루틴

- 학습하기(별 획득 기준) 5회

#### B. 상태 관리 루틴

- 쓰다듬기(젤로 터치) 5회
- `[과일/야채/베이크/음식/디저트]` 중 랜덤 카테고리 주기 3회
- 이닦기 2회
- 샤워시키기 2회
- 청소하기(똥 치우기) 2회
- 벌레 잡기 1회
- 잠자기 1회

### 3. 목표 선정 방식

- A 풀은 현재 1개이므로 고정 선택된다.
- B 풀에서는 매일 2개를 랜덤 선택한다.
- 음식 목표가 선택되면 음식 카테고리도 함께 랜덤 지정한다.
- 예시:
  - 학습하기(별 획득 기준) 5회
  - 디저트 주기 3회
  - 이닦기 2회

---

## 1회 인정 기준

- 학습하기: 게임 정상 종료 후 별을 획득한 경우 1회 인정
- 쓰다듬기: 젤로 터치 성공 시 1회 인정
- 음식 주기: 지정된 음식 카테고리의 아이템 먹이기 성공 시 1회 인정
- 이닦기: 이닦기 액션 성공 시 1회 인정
- 샤워시키기: 샤워 액션 성공 시 1회 인정
- 청소하기: 똥 1개를 제거하면 1회 인정
- 벌레 잡기: 벌레 1마리를 제거하면 1회 인정
- 잠자기: 수면 상태 진입 시 1회 인정(바로 깨워도 상관없음)

---

## 상태값 목표 제외 이유

초기 기획에서 검토했던 `포만감 80 이상`, `건강 80 이상` 같은 상태 달성형 목표는 1차에서 제외한다.

이유:

- 이미 달성된 상태에서 팝업만 열어도 자동 완료될 수 있다.
- 오늘 실제로 한 행동보다 기존 상태에 의해 보상이 결정될 수 있다.
- 일일 루틴의 목적이 `오늘의 플레이 유도`인데, 상태형 목표는 이 목적과 맞지 않는다.

따라서 1차 일일 루틴은 `행동 누적형 목표`만 사용한다.

---

## 로컬 저장 구조

일일 루틴의 진행도와 완료 상태는 로컬에만 저장한다.

예시:

```ts
{
  dateKey: "2026-03-20",
  tasks: {
    studyStars: { target: 5, progress: 2 },
    feedDessert: { target: 3, progress: 1 },
    brushTeeth: { target: 2, progress: 2 }
  },
  completed: false,
  claimed: false
}
```

### 로컬 저장 원칙

- 앱 진입 시 오늘 날짜 기준으로 루틴 상태를 로드한다.
- 날짜가 바뀌면 새 루틴을 생성한다.
- 각 목표는 행동 성공 시마다 로컬 진행도를 즉시 증가시킨다.
- 모든 목표 완료 시 `completed = true` 처리한다.
- 서버 claim 성공 시 `claimed = true` 처리한다.

---

## 서버 저장 최소안

서버에는 유저별 하루 1개의 claim 기록만 저장한다.

예시:

```ts
{
  userId: "abc123",
  dateKey: "2026-03-20",
  claimedAt: 1710860400000
}
```

### 서버 저장 원칙

- `reward` 정보는 1차에서 저장하지 않는다.
- 서버는 오늘 이미 claim 했는지만 판단한다.
- 랜덤 보상이더라도, 유저가 나중에 다시 확인할 필요가 없으면 저장하지 않는다.

---

## Claim 동작 방식

### 클라이언트

1. 로컬에서 목표 3개를 모두 달성한다.
2. 팝업에서 `보상 받기` 버튼이 활성화된다.
3. 버튼 클릭 시 서버리스 claim 요청을 보낸다.

### 서버

1. 로그인 유저인지 확인한다.
2. 오늘 `dateKey` 기준 claim 기록이 있는지 조회한다.
3. 이미 claim 했으면 보상을 지급하지 않는다.
4. 아직 claim 하지 않았다면 보상을 지급한다.
5. `claimedAt` 기록을 저장한다.

## 보상 정책

### 방향

- 보상은 `GRO + XP`를 함께 지급한다.
- 보상은 랜덤으로 지급한다.
- 보상량은 젤로의 현재 `evolutionStage`에 따라 다르게 설정한다.
- 일일 루틴은 메인 성장 수단이 아니라 `기분 좋은 일일 보너스`로 유지한다.

### 설계 원칙

- Stage가 높을수록 GRO와 XP 범위를 조금씩 상향한다.
- GRO는 체감되게, XP는 보조적으로 준다.
- 미니게임 메인 보상을 압도하지 않도록 수치는 보수적으로 잡는다.

이 수치는 초안이며 실제 밸런싱 시 조정 가능하다.

### 리밸런싱 기준

현재 성장 곡선과 미니게임 XP 기준을 반영해, 일일 루틴 XP는 `현재 단계의 적정 게임 0.5~0.8판 수준`으로 맞춘다.

- Stage 1 적정 XP 기준: 약 3 XP/game
- Stage 2 적정 XP 기준: 약 6 XP/game
- Stage 3 적정 XP 기준: 약 8~10 XP/game
- Stage 4 적정 XP 기준: 약 15 XP/game
- Stage 5 적정 XP 기준: 약 15~25 XP/game

따라서 일일 루틴은:

- 성장 보조 역할은 하지만, 메인 학습 루프를 대체하지 않는다.
- 매일 받으면 기분 좋은 수준의 XP만 지급한다.
- Stage가 올라갈수록 요구 성장량이 급격히 커지므로, XP와 GRO를 함께 완만하게 상향한다.

### 보상 의도

- Stage 1~2: 초반 적응 보조
- Stage 3: 본격 성장 구간의 작은 가속
- Stage 4~5: 성장량이 큰 구간이므로 GRO 체감은 올리고, XP는 과도하지 않게 유지

### 랜덤 보상 방식

일일 루틴 보상은 단계별로 `보통 / 상급 / 대박` 3단계 중 하나를 지급한다.

- 보통: 70%
- 상급: 25%
- 대박: 5%

이 구조의 목적은 다음과 같다.

- 대부분의 날에는 안정적인 보상을 제공한다.
- 가끔 상급 보상으로 만족감을 준다.
- 드물게 대박 보상으로 작은 기대감을 만든다.

### 단계별 기대값

확률형 지급 기준의 평균 기대값은 다음과 같다.

| Stage | 평균 GRO | 평균 XP |
| :--- | :--- | :--- |
| 1 | 약 14.6 | 약 2.35 |
| 2 | 약 21.9 | 약 3.35 |
| 3 | 약 34.9 | 약 4.75 |
| 4 | 약 49.3 | 약 7.10 |
| 5 | 약 66.0 | 약 9.05 |

이 기대값은 각 단계의 적정 미니게임 1판 보상보다 낮거나 비슷한 수준으로 유지해, 일일 루틴이 메인 성장 수단이 되지 않도록 설계한다.

## 서버 부하 관점

이 설계는 서버 사용량을 매우 낮게 유지한다.

유저 1명 기준 하루 최대:

- claim 조회 1회
- claim 기록 저장 1회
- 유저 재화/XP 업데이트 1회

즉, 서버 부하의 대부분은 `reward 저장 여부`보다 `claim API 호출 수`와 `DB 읽기/쓰기 횟수`에 의해 결정된다.

현재 구조에서는 `reward`를 저장하지 않아도 기능적으로 충분하며, 서버 사용 최소화 목표에도 부합한다.

## UI 흐름

### 진입 및 버튼 노출

1. Petroom 진입
2. FAB 버튼 영역 최상단에 `🗒️ 오늘 루틴` 버튼 노출

### 팝업 구성

버튼 클릭 시 일일 루틴 팝업을 연다.

- 팝업 타이틀: `Daily Routine`
- 오늘의 목표 3가지 표시
- 각 목표의 진행도와 달성 여부 표시
- 하단에 `보상 받기` 버튼 표시
- 미달성 상태에서는 `보상 받기` 버튼을 비활성화 상태로 유지

### 진행 및 보상 수령

1. 유저가 플레이를 진행하면 목표 진행도가 자동으로 증가한다.
2. 목표 3개를 모두 완료하면 `보상 받기` 버튼이 활성화된다.
3. 유저가 `보상 받기` 버튼을 누르면 claim 요청을 보낸다.
4. claim 성공 시 루틴 팝업은 닫힌다.
5. 이후 보상 내용 팝업을 노출한다.
6. 보상 내용 팝업은 기존 기차 보상 UI를 재활용한다.

### 수령 후 상태

- 일일 보상 수령 후 팝업을 다시 열면 하단 버튼은 `오늘 완료`로 표시한다.
- `오늘 완료` 상태에서는 버튼을 비활성화한다.

### UI 메모

- `오늘 루틴` 버튼은 상점 버튼보다 위에 배치한다.
- 목표는 텍스트만이 아니라 체크 상태가 함께 보여야 한다.
- 진행도는 예: `2 / 5` 형태로 직관적으로 노출한다.
- 보상 수령 전과 수령 후의 상태 변화가 명확히 보여야 한다.

## 구현 순서 제안

### 1. 데이터 및 상태 구조 정의

1. 로컬 일일 루틴 상태 모델 정의
2. 목표 타입 정의
3. A/B 목표 풀 정의
4. 음식 카테고리 랜덤 지정 방식 정의
5. 보상 테이블 및 Stage별 확률형 보상 정책 정의

### 2. 루틴 생성 및 날짜 전환 처리

1. 앱 또는 Petroom 진입 시 오늘 날짜 기준 루틴 로드
2. 로컬에 오늘 루틴이 없으면 새 루틴 생성
3. 날짜가 바뀌면 이전 루틴을 폐기하고 새 루틴 생성
4. claim 완료 상태도 날짜 기준으로 초기화

### 3. 진행도 반영 로직 연결

1. 학습하기(별 획득 기준) 성공 시 진행도 증가
2. 쓰다듬기 성공 시 진행도 증가
3. 음식 주기 성공 시 지정 카테고리와 일치하면 진행도 증가
4. 이닦기 성공 시 진행도 증가
5. 샤워 성공 시 진행도 증가
6. 똥 제거 시 진행도 증가
7. 벌레 제거 시 진행도 증가
8. 잠 푹자기 판정 시 진행도 증가

### 4. Petroom UI 추가

1. FAB 버튼 영역 최상단에 `🗒️ 오늘 루틴` 버튼 추가
2. 버튼 클릭 시 일일 루틴 팝업 오픈
3. 팝업에 `Daily Routine` 타이틀 적용
4. 목표 3개와 진행도, 달성 여부 표시
5. 하단 `보상 받기` 버튼 기본 비활성화 상태 적용

### 5. 완료 판정 및 보상 수령 UX

1. 목표 3개 완료 여부를 실시간 판정
2. 전부 완료 시 `보상 받기` 버튼 활성화
3. 수령 완료 후 버튼을 `오늘 완료` 상태로 전환
4. claim 성공 시 팝업 닫기
5. 보상 결과 팝업은 기존 기차 보상 UI 재활용

### 6. 서버리스 claim 연결

1. claim API는 진행도를 검증하지 않고 오늘 claim 여부만 확인
2. 서버에서 `evolutionStage` 기준 보상 등급 계산
3. `GRO + XP` 랜덤 보상 지급
4. 서버에는 `userId`, `dateKey`, `claimedAt`만 저장

### 7. 예외 처리

1. 중복 클릭 방지
2. claim 요청 중 로딩 상태 처리
3. 네트워크 실패 시 재시도 가능 상태 유지
4. 날짜 변경 직전/직후 루틴 재생성 처리
5. 이미 claim 완료된 날의 중복 지급 방지
6. 로컬 진행도는 완료 상태지만 서버 claim 실패한 경우 재청구 가능하도록 처리

### 8. 검증 및 밸런스 확인

1. 목표 진행 카운트가 의도대로 누적되는지 확인
2. 음식 카테고리 판정이 정확한지 확인
3. `잠 푹자기` 인정 조건 검증
4. Stage별 보상량이 과하지 않은지 확인
5. 하루 루틴 난이도가 과도하지 않은지 플레이 테스트

## 1차 출시 권장 범위

- 로그인 유저 기준으로 보상 claim 지원
- 일일 루틴 진행도는 로컬 전용
- 하루 1세트, 목표 3개
- A 풀 1개 + B 풀 2개
- 서버는 claim 여부만 저장
- 보상은 `GRO + XP` 랜덤 지급
- 보상량은 `evolutionStage` 기반 차등 적용
