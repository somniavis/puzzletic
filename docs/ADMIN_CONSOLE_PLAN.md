# Admin Console 분리 설계안

> 마지막 업데이트: 2026-04-07

## 개요

이 문서는 현재 서비스 내부에 섞여 있는 `디버그`, `개발 검수`, `운영 보조`, `내부 관리자용 UI`를
일반 사용자 화면에서 제거하고, 동일 서비스 내 별도 `Admin Console` 서브 페이지로 이관하기 위한 설계안입니다.

핵심 방향은 아래와 같습니다.

1. 사용자 UI에서 운영자 전용 기능을 제거한다.
2. 동일 도메인/동일 앱 안에서 `/admin` 계열 서브 페이지로 분리한다.
3. `admin` 권한 계정으로 로그인한 경우에만 설정에서 진입 버튼을 노출한다.
4. 버튼 노출 여부와 별개로, 실제 접근 권한은 서버/인증 계층에서 다시 검증한다.
5. 기존 디버그 기능은 점진적으로 어드민으로 이관하고, 사용자 경로에서는 제거한다.

이 방식은 별도 관리자 사이트를 새로 구축하는 것보다 구현 부담이 낮고,
현재 인증/라우팅/데이터 구조를 재사용하기 좋으며,
운영 기능이 유저 경험을 오염시키는 문제를 줄이는 데 적합합니다.

---

## 목표

이번 분리 작업의 목표는 아래와 같습니다.

1. 일반 사용자 화면에서 내부 운영용 요소를 완전히 숨긴다.
2. 어드민 계정만 접근 가능한 별도 관리 페이지를 제공한다.
3. 현재 서비스 내부에 숨어 있는 디버그/검수 기능을 중앙화한다.
4. 레이아웃 템플릿, 게임 템플릿, 특정 계정 상태 확인 등 운영 기능을 한곳에서 조회/제어 가능하게 한다.
5. 나중에 운영 기능이 늘어나더라도 확장 가능한 구조를 만든다.

---

## 왜 `/admin` 서브 페이지 방식이 좋은가

현재 서비스에 가장 현실적인 1차 해법은 `동일 URL 구조 안의 별도 라우트`입니다.

예시:

- `/admin`
- `/admin/users`
- `/admin/layouts`
- `/admin/games`
- `/admin/debug`

장점:

- 기존 로그인 세션과 인증 상태를 그대로 재사용 가능
- 신규 운영 도메인/배포 파이프라인 없이 빠르게 시작 가능
- 사용자 서비스와 같은 데이터 컨텍스트를 공유하기 쉬움
- 디버그 기능을 단계적으로 이관하기 좋음
- 필요 시 이후 별도 어드민 앱으로 독립시키기도 쉬움

주의:

- `설정에서 버튼을 숨기는 것`은 UX 분기일 뿐, 보안이 아님
- `/admin` URL 직접 접근을 반드시 서버/권한 계층에서 차단해야 함
- 어드민 분기 로직이 일반 사용자 컴포넌트 안에 과도하게 섞이지 않도록 분리해야 함

즉, 이 방식은 매우 적절하지만,
반드시 `UI 숨김 + 라우트 가드 + API 권한 검증`이 함께 가야 합니다.

## 제안 구조

### 1. 진입 방식

일반 사용자:

- 설정 메뉴에 `Admin` 버튼을 노출하지 않음

어드민 계정:

- 설정 메뉴에만 `Admin` 버튼 노출
- 버튼 클릭 시 `/admin` 진입

권장 동작:

1. 로그인 후 사용자 프로필 또는 custom claim 기반으로 `isAdmin` 판별
2. `isAdmin === true` 인 경우에만 설정 메뉴에 `Admin` 항목 렌더링
3. `/admin` 진입 시 프론트에서 한 번 확인
4. 어드민 전용 API 호출 시 서버에서 다시 확인

---

## 권한 모델 제안

초기에는 복잡한 권한 체계보다 아래 2단계로 시작하는 것이 좋습니다.

### 1. `user`

- 일반 사용자
- 어드민 UI 진입 불가
- 어드민 API 호출 불가

### 2. `admin`

- 운영자/마스터 계정
- 어드민 UI 진입 가능
- 조회형/조작형 기능 사용 가능

추후 필요 시 아래처럼 확장 가능합니다.

### 확장 권한 예시

- `viewer`: 조회만 가능
- `operator`: 운영 조작 가능
- `master`: 고위험 기능 포함 전체 접근 가능

초기 단계에서는 `admin` 단일 권한으로 시작하되,
고위험 기능은 별도 확인 절차를 두는 것이 안전합니다.

## 인증 및 보안 원칙

어드민은 일반 사용자 기능보다 더 강하게 보호해야 합니다.

### 필수 원칙

1. UI 노출 여부와 실제 권한 검증을 분리한다.
2. `/admin` 라우트는 프론트 라우터에서 보호한다.
3. 어드민 API는 서버에서 `admin` 여부를 반드시 재검증한다.
4. 모든 중요 조작은 감사 로그를 남긴다.
5. 고위험 작업은 읽기/쓰기 권한을 분리하거나 재확인 단계를 둔다.

### 권장 구현

- Firebase custom claims 또는 서버 측 관리자 allowlist 사용
- `ProtectedRoute`와 별도로 `AdminRoute` 추가
- `/api/admin/*` 계열 엔드포인트 분리
- 중요 요청 로그 필드 저장:
  - `adminUid`
  - `targetUid`
  - `action`
  - `payload summary`
  - `createdAt`

### 금지해야 할 방식

- 프론트에서만 `isAdmin` 체크하고 서버 검증 생략
- 숨겨진 버튼/환경변수만으로 관리자 기능 보호
- 일반 사용자 API에 `?debug=true` 같은 분기 추가

## 이관 대상 분류

현재 서비스 내부의 어드민성 기능은 성격별로 나눠서 옮기는 것이 좋습니다.

### 1. 조회형

운영자가 상태를 보기만 하는 기능입니다.

예시:

- 마이젤로 디버그 정보 조회
- 상점 내 랜드 탭의 랜드 레이아웃 템플릿 확인
- 게임 디버그 모드 내 레이아웃 템플릿 확인
- 특정 유저 상태/보유 데이터 조회
- 캐릭터/에셋/게임 메타 정보 조회

특징:

- 가장 먼저 옮기기 좋음
- 리스크가 낮음
- 어드민 UI 기반을 만드는 데 적합

### 2. 조작형

운영자가 특정 상태를 적용/변경하는 기능입니다.

예시:

- 특정 계정에 디버그 모드 부여
- 기능 플래그 강제 활성화
- 테스트 레이아웃 적용
- 특정 게임 템플릿 강제 로드

특징:

- 서버 검증 필요
- 누가 무엇을 바꿨는지 로그 필요

### 3. 고위험형

실서비스 데이터에 직접 영향을 주는 기능입니다.

예시:

- 재화 지급
- 진척도/레벨/상태 직접 수정
- 결제/프리미엄 상태 강제 변경
- 사용자 계정 복구/차단

특징:

- 초기 어드민 범위에서는 제외하거나
- 최소한 `master` 수준으로 제한하는 것이 좋음

## 권장 Admin IA

초기 어드민 정보 구조는 아래처럼 잡는 것을 권장합니다.

### `/admin`

- 운영 대시보드
- 최근 변경 이력
- 자주 쓰는 도구 바로가기

### `/admin/users`

- 사용자 검색
- 특정 사용자 상세 상태 조회
- 디버그 모드 부여/해제
- 실험 플래그/운영 플래그 적용

### `/admin/layouts`

- 랜드 레이아웃 템플릿 조회
- 게임 레이아웃 템플릿 조회
- 레이아웃 프리뷰
- 템플릿 메타 정보 확인

### `/admin/debug`

- 현재 서비스 내부에 흩어진 디버그 기능 이관
- 페이지별 디버그 도구 묶음 제공
- 운영자 전용 토글/시뮬레이터

### `/admin/content`

- 캐릭터/에셋/게임 메타 검수
- 개발/기획 확인용 리스트
- 향후 Character Admin 등 기존 관리 도구 통합 가능

### `/admin/logs`

- 관리자 액션 로그
- 최근 적용된 운영 변경 내역

## 사용자 UI에서 제거할 원칙

일반 사용자 화면에서는 아래 요소를 제거 대상으로 봅니다.

1. 디버그 모드 진입 버튼
2. 운영자만 의미 있는 토글
3. 개발 검수용 템플릿 프리뷰 UI
4. 테스트용 상태 강제 변경 버튼
5. 내부 운영 설명 문구 또는 상태 패널

제거 기준은 아래와 같습니다.

- 일반 사용자가 사용할 이유가 없는가
- 서비스 경험을 혼란스럽게 만드는가
- 내부 구조를 노출하는가
- 권한 없는 사용자가 접근하면 리스크가 있는가

이 기준에 해당하면 사용자 UI에서는 삭제하고,
동일 기능이 필요하면 어드민으로 옮기는 것이 맞습니다.

## 이관 우선순위

한 번에 전부 옮기기보다 아래 순서가 안전합니다.

### Phase 1. 진입 구조 분리

목표:

- `/admin` 라우트 생성
- `Admin` 버튼을 설정 메뉴에 조건부 노출
- `AdminRoute` 가드 도입

산출물:

- 어드민 페이지 셸
- 좌측 메뉴 또는 탭 구조
- 빈 화면이어도 권한/진입 구조 완성

### Phase 2. 조회형 기능 이관

목표:

- 현재 사용자 UI 내부의 조회형 디버그/템플릿 기능 이전

우선 추천 대상:

- 마이젤로 디버그 정보 보기
- 랜드 레이아웃 템플릿 보기
- 게임 레이아웃 템플릿 보기

이유:

- 읽기 전용이라 위험이 낮음
- 운영자 도구로서 가치가 바로 생김

### Phase 3. 조작형 기능 이관

목표:

- 특정 계정에 디버그 모드 부여
- 운영 플래그 적용
- 테스트 상태 부여

조건:

- 서버 권한 검증
- 액션 로그 저장
- 실패/성공 피드백 명확화

### Phase 4. 고위험 기능 검토

목표:

- 정말 필요한 항목만 제한적으로 추가

조건:

- 별도 승인 정책
- 더 강한 권한 구분
- 감사 로그 및 롤백 정책

## 데이터/기술 구조 제안

### 프론트엔드

- `src/pages/admin/*` 또는 `src/features/admin/*` 구조로 분리
- 일반 페이지와 어드민 페이지 컴포넌트 분리
- `AdminRoute` 또는 `RequireAdmin` 가드 도입
- 설정 메뉴에서만 `Admin` 링크 노출

### 백엔드

- 일반 사용자 API와 분리된 `/api/admin/*` 엔드포인트 구성
- 모든 어드민 요청에서 권한 재검증
- 고위험 작업은 별도 액션 로그 기록

### 상태 관리

- `AuthContext` 또는 유사 전역 상태에서 `isAdmin` 제공
- 가능하면 로그인 시점에만 권한 판별 후 캐시
- 권한 변경 시 강제 재동기화 경로 마련

---

## 기존 기능과의 관계

현재 이미 관리자 성격을 가지는 기능이나 페이지가 있다면,
이후에는 아래 원칙으로 재배치하는 것이 좋습니다.

1. 개발/기획 검수 목적 페이지는 `/admin/content` 또는 `/admin/debug`로 이동
2. 특정 유저 조작 기능은 `/admin/users`로 이동
3. 템플릿/레이아웃 확인 도구는 `/admin/layouts`로 이동
4. 일반 라우트에서 직접 열리던 관리자 페이지는 점진적으로 비노출 처리

예를 들어 기존 `CharacterAdmin` 같은 페이지는
장기적으로 `/admin/content/characters` 또는 `/admin/gallery/characters` 성격으로 정리하는 편이 자연스럽습니다.

## UX 원칙

어드민은 기능이 많아지기 쉬우므로 처음부터 구조를 단순하게 잡는 것이 중요합니다.

### 권장 UX

- 좌측 사이드바 또는 상단 탭 기반 구조
- 조회형과 조작형을 시각적으로 구분
- 위험 액션은 별도 경고 스타일 사용
- 사용자용 UI 톤과는 구분되지만, 디자인 시스템은 재사용

### 피해야 할 UX

- 일반 페이지 안에 숨겨진 long-press 디버그 진입
- 한 화면에 운영 기능을 전부 몰아넣는 방식
- 동일 버튼으로 조회/수정/적용을 섞는 구조

## 운영 리스크와 대응

### 리스크 1. 숨김 UI를 보안으로 오해

대응:

- 서버에서 어드민 권한 재검증
- 프론트와 API 모두 분리

### 리스크 2. 일반 코드에 어드민 분기문 확산

대응:

- `/admin` 라우트와 전용 컴포넌트로 격리
- 일반 사용자 화면에서는 관련 컴포넌트 제거

### 리스크 3. 운영자가 실수로 데이터 변경

대응:

- 조회형 우선 이관
- 조작형부터는 로그 저장
- 고위험 기능은 늦게 열기

## 추천 결론

현재 서비스에는 아래 방향이 가장 적절합니다.

1. 어드민 기능은 별도 외부 사이트가 아니라 동일 앱의 `/admin` 서브 페이지로 시작한다.
2. 어드민 계정으로 로그인한 경우에만 설정 메뉴에 `Admin` 버튼을 노출한다.
3. 기존 사용자 화면에 숨어 있는 디버그/검수 UI는 점진적으로 제거한다.
4. 제거한 기능은 조회형부터 어드민으로 이관한다.
5. 조작형 기능은 서버 검증과 로그 체계를 갖춘 뒤 추가한다.

즉, 이번 어드민 설계의 핵심은
`숨겨진 디버그 모드`를 계속 늘리는 것이 아니라,
`권한 있는 운영자만 접근 가능한 정식 관리 콘솔`로 재구성하는 것입니다.

## 바로 이어서 할 작업 제안

## 개발 착수 메모

이번 라운드의 실제 착수 범위는 아래와 같습니다.

### 1차 구현 범위

1. `AuthContext` 에 `isAdmin` 추가
2. 설정 메뉴에서 어드민 계정만 `Admin` 버튼 노출
3. `/admin` 라우트 및 `AdminRoute` 추가
4. 기존 `Gallery`, `Stats`, `Layout Preview` 를 어드민 라우트로 재배치
5. 프로필 페이지 내부 디버그 패널 제거

### 임시 권한 방식

초기 구현은 서버 기반 관리자 권한 시스템이 아직 없으므로
`VITE_ADMIN_EMAILS` 환경변수 기반 이메일 allowlist로 시작합니다.

형식 예시:

```env
VITE_ADMIN_EMAILS=master@example.com,ops@example.com
```

주의:

- 이 방식은 1차 프론트 보호용입니다.
- 최종적으로는 Firebase custom claims 또는 서버 검증으로 이동해야 합니다.

### 다음 구현 우선순위

1. `/admin/debug` 화면에 프로필 디버그 액션 재배치
2. 랜드 레이아웃 템플릿 조회 UI 추가
3. 게임 레이아웃 템플릿 조회 메뉴 확장
4. 특정 사용자 검색/디버그 적용 기능 설계
5. `/api/admin/*` 계열 서버 검증 설계

## 마스터 어드민 계정 생성 방식

현재 구현 기준으로는
`어드민 전용 회원가입 플로우`나
`콘솔에서 새 마스터 계정을 자동 생성하는 기능`은 아직 없습니다.

현재는 아래 방식으로 마스터 어드민 계정을 지정합니다.

### 현재 방식

1. Firebase Authentication 에 일반 로그인 계정을 먼저 생성
2. 그 계정 이메일을 `VITE_ADMIN_EMAILS` 환경변수에 추가
3. 해당 계정으로 로그인하면 `isAdmin = true`
4. 설정 메뉴에 `Admin` 버튼 노출
5. `/admin` 라우트 접근 가능

즉, 현재 마스터 어드민 생성은
`새 관리자 계정을 시스템 안에서 만드는 것`이 아니라,
`이미 존재하는 Firebase 계정을 어드민으로 승격하는 방식`입니다.

### 환경변수 형식

```env
VITE_ADMIN_EMAILS=master@example.com,ops@example.com
```

규칙:

- 쉼표로 여러 이메일 지정 가능
- 이메일 비교는 소문자 기준으로 처리
- 로그인한 Firebase 계정 이메일과 일치하면 어드민으로 판별

### 현재 코드 반영 위치

- `src/contexts/AuthContext.tsx`

동작 요약:

1. 로그인된 사용자 이메일 확인
2. `VITE_ADMIN_EMAILS` 목록과 비교
3. 일치하면 `isAdmin` true 설정

## 마스터 어드민 계정 생성 절차

초기 운영에서는 아래 순서를 권장합니다.

### 1. Firebase에 관리자 계정 생성

예시:

- `master@yourdomain.com`

방법:

- Firebase Console > Authentication 에서 직접 계정 생성
- 또는 일반 회원가입 플로우로 계정 생성 후 이후 승격

### 2. 환경변수에 관리자 이메일 등록

로컬 또는 배포 환경에 아래 추가:

```env
VITE_ADMIN_EMAILS=master@yourdomain.com
```

### 3. 앱 재빌드 또는 재배포

Vite 환경변수는 빌드 시 반영되므로,
환경변수 변경 후에는 개발 서버 재시작 또는 재배포가 필요합니다.

### 4. 관리자 계정으로 로그인하여 확인

확인 항목:

- 설정 메뉴에 `Admin` 버튼 표시
- `/admin` 진입 가능
- 일반 계정으로는 버튼 비노출
- 일반 계정으로 `/admin` 직접 접근 시 차단

---

## 현재 방식의 한계

이 방식은 빠르게 시작하기에는 좋지만,
최종 운영 보안 모델로는 부족합니다.

이유:

1. 프론트 환경변수 기반 판별이라 관리가 거칠다.
2. 권한 변경 시 재배포가 필요하다.
3. 서버 API 차원의 강한 권한 검증 모델이 아니다.
4. `master`, `operator`, `viewer` 같은 세부 권한 분리가 어렵다.

즉, 지금 방식은
`1차 어드민 콘솔 진입용 임시 게이트`
로 보는 것이 맞습니다.

---

## 운영 단계 권장 업그레이드

운영 오픈 전 또는 직후에는 아래 구조로 올리는 것을 권장합니다.

### 권장 최종 방식

1. Firebase custom claims 로 `admin: true` 부여
2. 필요 시 `role: master | operator | viewer` 형태로 세분화
3. 프론트는 claim 기반으로 `isAdmin` 판별
4. 서버 `/api/admin/*` 도 동일 claim 을 재검증
5. 모든 중요 액션은 감사 로그 저장

### 권장 이유

- 프론트/서버 권한 모델 일치
- 계정별 역할 관리 용이
- 환경변수 재배포 없이 권한 변경 가능
- 특정 계정 상태 변경 같은 운영 기능을 안전하게 확장 가능

## 현재 결론

현 시점의 마스터 어드민 생성 방법은 아래로 정리합니다.

1. Firebase 로그인 계정을 만든다.
2. 그 이메일을 `VITE_ADMIN_EMAILS` 에 넣는다.
3. 해당 계정으로 로그인하면 어드민 콘솔에 접근한다.

그리고 이 방식은 어디까지나
`빠른 초기 운영용`
이며,
향후에는 `Firebase custom claims + 서버 검증` 방식으로 전환하는 것이 목표입니다.

### 현재 지정 계정 메모

현재 테스트/운영 마스터 어드민으로 사용할 계정은 아래로 통일합니다.

- `grogrojello.dev@gmail.com`

운영 원칙:

- 기존 `master@grogrojello.com` 계정은 마스터 어드민 allowlist 에서 제거
- 테스트/운영 환경 모두 `VITE_ADMIN_EMAILS=grogrojello.dev@gmail.com` 기준으로 맞춤
- 로컬 개발 환경도 동일 이메일 기준으로 맞춤
